# GitHub Open Source Site Rework

## Overview

在用户明确要求“不要收费”“改为开源获取”“直接引导去 GitHub 仓库”时，使用这个技能。将这类需求视为一次**完整的产品定位重写**，而不是单纯改一个按钮链接；需要同步处理前端区块、弹窗逻辑、后端试用耗尽提示、测试、检查点与最终交付，避免页面表面改了，但代码和文案里仍残留收费路径。

## Workflow Decision

如果用户只要求替换单个按钮 URL，可以做最小改动，但仍然必须做一次全局残留检索。如果用户要求移除收费、取消升级转化、删除价格区、把试用耗尽提示改为仓库引导，执行下方**完整流程**。

## Full Workflow

第一步，先登记工作。更新任务计划，并在当前项目根目录的 `todo.md` 中追加未完成事项，再开始实现。

第二步，定位所有收费触点。优先检查首页或主落地页，然后全局检索与收费相关的业务词，例如 `升级`、`付费`、`收费`、`定价`、`pricing`、`paywall`、`联系升级`、`查看升级方案`。开始修改前，先阅读 `references/residue-checklist.md`。

第三步，改写前端导向。移除或改写价格区、升级弹窗、付费 CTA、试用结束后引导，以及底部二次转化区。统一替换为 GitHub 仓库、README、部署说明或开源项目获取引导。主转化位通常保留在 Hero、试用结果区、试用耗尽状态和底部获取区。

第四步，同步后端提示。只要后端存在试用次数耗尽、额度锁定、配额用尽等提示，也要改成 GitHub 仓库引导，而不是保留“升级套餐继续使用”之类旧措辞。

第五步，做残留清扫。再次对全项目执行全局检索，逐个核查命中项，不要假设任何命中都是无害的。只有与通用 UI 组件内部实现无关、且不暴露给产品用户的内容，才可以保留。

第六步，补充回归测试。必要时先提取可复用常量，再新增或更新 Vitest 测试，确保新的提示文案包含 `GitHub`、`开源`、`仓库` 等关键词，并且不再包含 `升级`、`付费`、`套餐` 等旧词。

第七步，做实现验证。先跑本次新增或受影响的测试，再执行构建。代码稳定后再检查预览状态，确认页面可正常访问、关键区块已替换、试用链路提示已一致。

第八步，完成交付。把本次任务在 `todo.md` 中标记为完成，保存检查点，并用简短总结说明“收费导向已移除、现统一引导至 GitHub 仓库”。随后给出 2 至 3 条紧贴仓库引导的下一步建议，例如补 README 快速开始、增加复制结果按钮、增加架构图。

## Editing Standards

使用直接、明确的引导措辞，优先写成“查看 GitHub 仓库”“获取完整开源项目”“查看 README 与部署说明”。避免留下模糊过渡语，例如“了解更多套餐”“继续升级”或“联系销售”。如果原有价格区在版式上承担了重要布局作用，可以改造成“开源获取”“仓库内容”“快速开始”区块，而不要简单删空导致页面结构塌陷。

## Testing Standards

每次至少验证三件事：第一，页面可以成功构建；第二，试用耗尽或配额锁定路径不再出现收费升级措辞；第三，全局检索后，产品页面和服务端业务提示中不再残留面向用户的收费文案。具体检索模式和验收标准见 `references/residue-checklist.md`。

## Deliverables

默认交付物是更新后的站点检查点。如果用户要求把这套流程本身沉淀为技能，则交付 `skills/github-open-source-site-rework/SKILL.md`。

## Resources
