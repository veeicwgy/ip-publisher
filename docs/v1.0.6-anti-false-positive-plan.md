# IP Publisher v1.0.6 去误报修复方案

## 一、问题判断

当前 `ip-publisher` 在 ClawHub / OpenClaw 被标记为 **suspicious**，核心不是“发现恶意代码”，而是**声明内容、打包内容与平台可见元数据三者不一致**。现有线上说明显示，扫描器认为该 skill 自称提供脚本、内部模块、本地文件读写与网页抓取能力，但实际被扫描到的发布包更像只有 `SKILL.md` 与 `prompt.md` 的 instruction-only 包，因此触发了 capability mismatch。

这次 `v1.0.6` 的目标，不是继续加强营销文案，而是先把 **“安装包里到底有什么”**、**“默认会不会碰本地文件”**、**“默认会不会抓网页/发布”** 三件事写清楚、收窄、做实。只要让技能声明和实际包内容重新对齐，误报概率就会明显下降。

## 二、最可能触发误报的四个点

| 风险点 | 当前表现 | 为什么会触发误报 | v1.0.6 修复原则 |
| --- | --- | --- | --- |
| 打包内容不一致 | 已发布 skill 包主要可见 `SKILL.md` 与 `prompt.md`，但正文提到 `scripts/quickstart.py`、`scripts/generate-publish-pack.py`、多个内部模块 | 扫描器会把它理解为“声称有实现，但 manifest 里没有” | 不再把仓库脚本写成已安装 skill 自带资产；如要保留，必须放进 skill 目录内并明确为 bundled resources |
| 本地文件边界不清 | 多处出现 `~/.ip-publisher/profile.yaml` 读取/写入 | 这会被视为可影响本地状态的敏感行为 | 改成默认只使用会话内资料；只有用户明确要求“导出/保存”时才说明可选本地落盘 |
| 网络抓取边界不清 | 文案里出现热点抓取、公开网页抓取/搜索，但未把默认条件写清 | 扫描器会把它视为未明确约束的外部网络行为 | 改成“默认不主动抓取网络；只有用户明确要求最新热点时才执行检索，并在结果中标注来源” |
| 自动发布预期过强 | 虽然正文已收敛，但仍有“一键”“发布执行”“后续自动发布”等表述 | 平台会把发布、插件、外部账号结合看作更高风险动作 | 改成“生成三平台改写结果与可审阅发布包”；删除未来自动发布承诺和插件暗示 |

## 三、v1.0.6 的直接修改方案

### 1. 优先修改 `skills/ip-publisher/SKILL.md`

这个文件是误报修复的核心入口。当前版本最需要删除的，不是平台词，而是**会让扫描器以为包内自带脚本和模块**的写法。

| 文件 | 需要删除或改写的内容 | 建议替换为 |
| --- | --- | --- |
| `skills/ip-publisher/SKILL.md` frontmatter `description` | “一键生成发布包”“Use for: ... publish pack ... 默认不自动登录...” 这种营销 + 能力混写方式 | 改成更克制的结果描述：`帮你把一个话题改写成小红书、公众号、知乎三个版本，并整理成可审阅的发布包。Use for: xiaohongshu rewrite, wechat article draft, zhihu draft, publish pack.` |
| `What users get after install` | “仓库里还提供 scripts/quickstart.py 和 scripts/generate-publish-pack.py” | 改成“已发布 skill 默认只提供三平台改写与发布包整理工作流；仓库脚本属于 companion repo 资源，不属于安装后自动可见资产” |
| `External dependencies and declared interfaces` 整表 | `ip-profile`、`hotspot-fetcher`、`article-generator`、`humanizer`、`cover-generator`、`multi-publisher` 被写成已存在模块 | 改成 “Workflow stages” 表，描述为“人设整理、选题确认、平台改写、润色、封面建议、发布包整理” 六个步骤，不再伪装成运行时模块 |
| `Safety and operational boundaries` | `默认不自动保存...仅在用户明确同意时写入 ~/.ip-publisher/profile.yaml` | 进一步收窄为“已发布 skill 默认不读取、不写入本地文件；如用户明确要求导出结构化人设，可在本地仓库模式下另行执行” |
| `Step 1 - 确认是否已有话题与平台` | “如果没有话题，再补做热点抓取与选题建议” | 改成“如果没有话题，先向用户要主题方向；只有用户明确要求最新热点时，才进入检索并标注来源” |
| `Step 6 - 生成发布包` | “若需要结构化文件，可调用 scripts/...；若用户明确要求执行发布，则先检查 multi-publisher...” | 改成“默认输出对话内可审阅发布包；结构化文件与发布执行属于仓库模式或外部工具模式，不是已安装 skill 的默认能力” |

### 2. 同步修改 `skills/ip-publisher/prompt.md`

`prompt.md` 必须和 `SKILL.md` 保持严格一致，否则哪怕前台 description 变了，运行时口径仍可能被判定为能力越界。

| 当前问题 | v1.0.6 改法 |
| --- | --- |
| 首句把 skill 描述成主流程执行器，并默认包含热点、人设、本地配置、封面和发布准备 | 改成“你是三平台内容改写助手”，默认任务只有：确认主题、生成 Xiaohongshu / WeChat / Zhihu 三个平台版本、整理可审阅发布包 |
| 明写“只有在用户明确允许时，才读取或写入 `~/.ip-publisher/profile.yaml`” | 改成“默认不读写本地文件；若用户主动要求导出配置，再单独说明该动作属于本地仓库模式” |
| “如果没有话题，再补做热点抓取” | 改成“优先让用户给主题；只有用户明确要求实时热点时，才做检索并引用来源” |
| “只有在用户明确要求且本地发布条件已满足时，才进入发布执行” | 彻底删除“发布执行”路径，只保留“生成可人工发布的发布包” |

### 3. 收敛 `skills/ip-profile/*`

`ip-profile` 当前把“保存到 `~/.ip-publisher/profile.yaml`”写成目标本身，这对扫描器来说太像持久化写盘型 skill。v1.0.6 应把它改成“结构化整理用户人设”，而不是“默认写文件”。

| 文件 | 需要调整的点 | 修改方向 |
| --- | --- | --- |
| `skills/ip-profile/SKILL.md` | `Goal` 直接写“保存到 ~/.ip-publisher/profile.yaml” | 改成“输出结构化人设 YAML 与摘要；仅在用户明确要求导出时才说明可保存到本地文件” |
| `skills/ip-profile/prompt.md` | 首句无条件要求“整理并保存到 ~/.ip-publisher/profile.yaml” | 改成“整理成 YAML；默认在对话中返回，不主动保存到本地路径” |

### 4. 收敛 `skills/multi-publisher/SKILL.md`

这个子技能当前最大的风险，不是现在做了危险操作，而是还保留了**Wechatsync 插件依赖**和**后续真正自动发布**的承诺。对平台扫描来说，这会放大“发布 + 插件 + 外部账号”的风险印象。

| 当前问题 | v1.0.6 改法 |
| --- | --- |
| “多平台推送依赖 Wechatsync 浏览器插件” | 改成“该 skill 只负责整理多平台可复制发布包，不默认依赖任何浏览器插件或第三方发布器” |
| “后续计划：接入平台 API 实现真正自动发布” | 删除整句 |
| `Goal` 中“必要时再调用仓库内脚本” | 改成“默认输出对话内结果；如在仓库模式下工作，可再使用 companion scripts 导出文件” |

## 四、文件结构层面的修复建议

如果你还想保留“仓库里确实有脚本”这层价值，最稳妥的方式不是在 ClawHub 已发布 skill 正文里反复提它，而是把**包内资源**和**仓库 companion 资源**彻底分开。

| 目标 | 建议结构 | 目的 |
| --- | --- | --- |
| 让已发布 skill 自洽 | `skills/ip-publisher/{SKILL.md,prompt.md,references/...}` | 让扫描器看到安装包本身就是完整闭环 |
| 保留 repo 里的脚本价值 | 把 `scripts/quickstart.py`、`scripts/generate-publish-pack.py` 视为 companion repo，不写成“安装后自动拥有” | 避免 capability mismatch |
| 明确边界 | 在 `skills/ip-publisher/references/` 下新增 `package-boundary.md` | 用一页短文明确“已发布 skill 默认只交付对话工作流，仓库脚本是额外仓库资源” |
| 降低 instruction-only 误判 | 在 skill 目录内放 1–2 个 reference 文件，而不是只剩两份主文件 | 让包结构更完整，也减少“只有提示词”的观感 |

## 五、建议新增的 reference 文件

v1.0.6 不一定要把 repo 脚本复制进 skill 包，但至少应增加两个短 reference 文件，让 skill 包呈现出更清晰的边界与结果契约。

| 新文件 | 作用 | 应写什么 |
| --- | --- | --- |
| `skills/ip-publisher/references/package-boundary.md` | 说明安装包内和仓库内各自包含什么 | 明写“ClawHub 发布包默认包含 SKILL.md、prompt.md 与 references；仓库脚本不属于已安装 skill 资产” |
| `skills/ip-publisher/references/output-contract.md` | 固定最终交付格式 | 只写三平台标题、正文、标签建议、封面建议、发布说明这几个输出字段，避免泛化成更大的自动化系统 |

## 六、v1.0.6 文案原则

文案上最重要的不是“更会卖”，而是**让每一句都可被平台扫描器验证**。因此建议统一使用下面三条原则。

| 原则 | 保留 | 删除 |
| --- | --- | --- |
| 结果可验证 | “改写成三平台版本”“整理成可审阅发布包” | “一键代发”“全流程自动化” |
| 默认边界明确 | “默认只在对话内输出结果” | “默认读取/写入本地配置”“默认抓热点”“默认进入发布执行” |
| 资产归属清楚 | “仓库脚本是 companion repo 资源” | “安装后自带脚本/模块” |

## 七、重新发布 v1.0.6 的顺序

v1.0.6 不建议只改 description 就再发一次。更稳妥的顺序是：先收敛 skill 包，再补 references，再发版，再去评论 issue #1714 说明已经主动修复 capability mismatch。

| 顺序 | 动作 | 验收标准 |
| --- | --- | --- |
| 1 | 修改 `skills/ip-publisher/SKILL.md` 与 `prompt.md` | 不再出现 repo 根脚本、运行时模块、默认本地读写、默认发布执行承诺 |
| 2 | 修改 `skills/ip-profile/*` 与 `skills/multi-publisher/SKILL.md` | 不再把本地保存和未来自动发布写成核心目标 |
| 3 | 新增 `references/package-boundary.md` 与 `references/output-contract.md` | skill 包不再是“只有两份主文件”的 instruction-only 观感 |
| 4 | 重新发布 v1.0.6 | 详情页 summary、display name、tags 与新口径一致 |
| 5 | 在 `openclaw/clawhub#1714` 追评 | 告知平台已移除 capability mismatch 来源，请求复扫与撤销 suspicious |

## 八、建议采用的 v1.0.6 口径

如果你要一个可直接落地的主描述，我建议把主 skill description 收敛成下面这句：

> 帮你把一个话题改写成小红书、公众号、知乎三个版本，并整理成可审阅的发布包。默认只输出对话内结果，不自动读取本地文件、不默认抓取网络、不执行代发布。

这句的好处是三点同时成立：**用户能看懂价值、扫描器能看懂边界、已发布包也能真正承接这个承诺。**
